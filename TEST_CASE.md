# 测试用例

## 测试场景

**输入需求**：分析 Transformer 架构的创新点

## 测试步骤

### 1. 命令行测试

```bash
# 测试 API 连接
python main.py --test

# 预期输出：
# 正在测试 API 连接...
# ✅ 连接成功！模型: gpt-4o

# 运行分析
python main.py "请分析《Attention Is All You Need》论文中 Transformer 架构的创新点，包括其在自然语言处理领域的贡献和后续影响"
```

### 2. Web 界面测试

1. 启动服务：`python server.py`
2. 浏览器打开：`http://localhost:8000`
3. 点击「API配置」，填入 API Key 并测试连接
4. 选择「论文分析」模式
5. 点击示例「论文」按钮填入示例内容
6. 点击「开始分析」

## 预期输出结构

### Agent 1: Coordinator（分析计划）

```
## 输入分析
- **类型**：论文全文
- **领域**：自然语言处理 / 深度学习
- **关键词**：Transformer, 自注意力机制, 编码器-解码器, 位置编码, 并行计算

## 分析计划

### 分析专员（Analyzer）重点任务
- 重点分析 Transformer 架构的核心组件（自注意力机制、多头注意力、位置编码）
- 分析与传统 RNN/LSTM 架构的对比实验设计
- 关注模型在机器翻译任务上的性能指标

### 评审专员（Reviewer）重点任务
- 评估自注意力机制的创新性
- 分析方法论的严谨性（实验设置、数据集选择）
- 关注模型的计算效率和可扩展性
```

### Agent 2: Analyzer（内容分析）

```
## 内容概述
- 研究背景：序列到序列任务依赖 RNN/CNN，存在长距离依赖和并行化问题
- 核心问题：如何设计完全基于注意力的架构
- 研究目标：提出 Transformer 取代 RNN/CNN

## 方法论分析
- 研究方法：提出新架构 + 实验验证
- 数据来源：WMT 2014 英德/英法翻译数据集
- 分析框架：BLEU 评分对比

## 主要发现
- Transformer 在英德翻译上达到 28.4 BLEU，超越所有已有模型
- 训练时间大幅缩短（3.5天 vs 数周）
- ...

## 数据评估
- 数据集选择合理（标准基准数据集）
- 消融实验设计完善
```

### Agent 3: Reviewer（批判性评审）

```
## 创新性评估
- **理论创新**：首次证明纯注意力架构可以完全替代 RNN/CNN
- **方法创新**：多头自注意力机制、位置编码方案
- **应用创新**：为后续 BERT、GPT 系列奠定基础

## 方法论评审
- 实验设计合理，包含消融实验
- BLEU 评价指标单一，缺乏多样性评估

## 局限性分析
- 仅在机器翻译任务上验证
- 对超长序列的计算复杂度问题（O(n²)）
- 缺乏对其他 NLP 任务的泛化验证

## 改进建议
- 扩展到更多 NLP 任务
- 研究长序列优化方案
```

### Agent 4: Synthesizer（综合报告）

```
# 综合分析报告

## 研究概述
（整合 Coordinator 和 Analyzer 的分析...）

## 核心发现与贡献
（综合分析结果...）

## 方法论评价
（结合 Analyzer 和 Reviewer 的评价...）

## 创新性分析
（基于 Reviewer 的评估...）

## 问题与局限
（基于 Reviewer 的分析...）

## 综合评价
（整体学术评价...）
```

### Agent 5: Advisor（研究建议）

```
## 后续研究方向
- 长序列注意力优化（如 Sparse Attention、Linear Attention）
- 多模态 Transformer 应用
- 高效 Transformer 架构搜索

## 方法改进建议
- 引入更多评价指标（如语义相似度）
- 扩展实验到更多语言对

## 实际应用建议
- 预训练模型（BERT、GPT 系列）
- 跨领域迁移学习

## 资源与可行性
- 计算资源需求评估
- 时间规划建议

## 总结展望
- Transformer 已成为 NLP 基础架构
- 向多模态和通用人工智能发展
```

## 验证要点

1. **Agent 协作链**：每个 Agent 是否正确接收前序 Agent 的输出
2. **职责分离**：各 Agent 的输出是否有明确的职责边界（不重叠）
3. **结果映射**：每个 Agent 的输出是否正确映射到对应的 Tab
4. **错误处理**：API Key 错误/网络超时/服务不可用的处理
5. **流式推送**：SSE 是否实时推送每个 Agent 的状态变化
